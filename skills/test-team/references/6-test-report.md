
# 测试报告生成 Agent

## 工作流程

### 步骤 1：读取数据

按顺序读取（用 Read 工具读 .md，用脚本读 .json）：
1. 需求文档 `**/REQ-*.md`
2. 用例 `**/TC-*.json`（`case_json_manager.py read`）
3. 执行记录 `**/EXEC-*.json`（`exec_json_manager.py read --type exec`）
4. 缺陷记录 `**/BUGS-*.json`（`exec_json_manager.py read --type bug`，如存在）
5. 评审报告 `**/*-review-*.md`（如存在）

### 步骤 2：数据分析

- **执行统计**：总数/执行数/通过/失败/阻塞/跳过，通过率=通过/执行×100%
- **需求覆盖**：每需求的用例数/执行数/通过数
- **缺陷统计**：按严重程度/模块/状态分布，缺陷密度=缺陷数/用例数
- **一致性检查**：总数=各状态之和，缺陷数与失败用例一致

### 步骤 3：质量评估

| 评级 | 条件 |
|------|------|
| 优秀 | 通过率≥95%，无严重缺陷 |
| 良好 | 通过率≥90%，严重缺陷≤2 |
| 一般 | 通过率≥80%，严重缺陷≤5 |
| 较差 | 通过率<80% 或严重缺陷>5 |

稳定性：是否有崩溃/致命缺陷/阻塞用例

### 步骤 4：风险识别

未修复严重缺陷 / 阻塞用例 / 覆盖不足的需求 / 时间压力导致的测试不充分

### 步骤 5：生成报告

保存到 `{PROJECT_DIR}/reports/REPORT-YYYYMMDD-HHMM.md`，格式：

```markdown
# 测试报告
**报告编号** / **项目名称** / **测试周期** / **报告日期**

## 测试执行摘要
| 指标 | 数值 | 说明 |
用例总数 / 执行数 / 通过数 / 失败数 / 阻塞数 / 通过率

## 缺陷分析
按严重程度/模块分布 + 典型缺陷列表

## 质量评估
功能质量评级 + 稳定性评级 + 用户体验评级

## 遗留风险
| 风险编号 | 描述 | 影响程度 | 建议 |

## 测试结论
整体评估：通过/有条件通过/不通过
上线建议：建议上线/修复后上线/不建议

## 改进建议
短期（本版本）+ 长期（下版本）

## 附录
引用文档路径 + 测试数据 + 测试团队
```

### 步骤 6：测试结论

- ✅ 通过（建议上线）：通过率≥95%，无严重缺陷，无阻塞，核心功能全通过
- ⚠️ 有条件通过（修复后上线）：通过率≥85%，严重≤2，阻塞≤2
- ❌ 不通过：通过率<85% / 致命缺陷 / 核心不可用 / 阻塞>2

### 步骤 7：输出反馈

告知用户：报告路径 + 通过率 + 缺陷数 + 测试结论 + 上线建议

## 质量标准

- 统计数据准确（总数=各状态之和）
- 通过率计算正确
- 缺陷数与执行记录一致
- 引用路径存在
- 结论与数据匹配

## 报告自检

生成后检查：数据一致性 / 通过率正确 / 缺陷数一致 / 引用路径存在 / 结论匹配
异常时提示用户检查执行记录完整性。
