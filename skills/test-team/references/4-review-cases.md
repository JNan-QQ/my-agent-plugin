
# 测试用例评审 Agent

## 评审维度

1. **需求覆盖度**：每个需求是否有对应用例
2. **场景完整性**：正常/异常/边界是否覆盖
3. **用例有效性**：步骤清晰、预期结果明确
4. **冗余控制**：是否存在重复用例

## 评审意见写法规范（必须遵守）

### 引用用例规则

每条评审意见必须引用用例原文，格式：

> **TC005** 引用：
> - title: "验证登录功能"
> - steps: "1. 操作登录页面 2. 测试各种情况"
> - expected: "显示成功"
> **问题**：标题太粗（"登录功能"含多个测试点），步骤无具体数据，预期不可验证
> **建议**：拆分为TC005a（正确密码登录）和TC005b（错误密码登录），步骤写具体输入值

### 问题分级标准

| 级别 | 定义 | 示例 |
|------|------|------|
| 🔴 严重 | 步骤不可执行 / 预期不可验证 / 关键需求无覆盖 | 步骤写"输入正确数据"（无具体值）；预期写"正常"（不可验证） |
| 🟡 一般 | 标题模糊 / 前置条件不足 / 优先级不当 | 标题写"验证登录功能"（太粗）；前置条件缺少测试数据 |
| 🟢 建议 | 格式优化 / 补充信息 / 步骤可精简 | 建议补充网络环境要求；步骤3和4可合并 |

### 冗余识别规则

以下情况判定为冗余：

| 冗余类型 | 判定标准 | 示例 |
|---------|---------|------|
| 标题雷同 | 两条用例测试同一件事 | TC001"验证错误密码提示" vs TC002"验证密码不正确提示" |
| 步骤重复 | 步骤序列相同，仅数据不同 | TC003和TC015都是"打开登录→输入admin→输入密码→点击登录"且预期相同 |
| 预期相同 | 不同用例的expected字段完全相同 | TC008和TC012预期都是"显示错误提示" |

### 评审意见模板

```
**[严重程度] [问题类型] TCxxx**
引用：
- title: "[用例标题]"
- steps: "[步骤摘要]"
- expected: "[预期摘要]"
问题：[具体说明]
建议：[给出修改方案]
```

## 工作流程

### 步骤 1：读取文件

1. Glob 查找 `**/TC-*.json`（path: {PROJECT_DIR}），选最新
2. 分批读取（每次 15-20 个）：
```bash
python {SKILL_DIR}/scripts/case_json_manager.py read --file {TC_PATH} --start 0 --end 20
```
3. 读取需求文档 `**/REQ-*.md`
4. 调 analyze 获取覆盖度+质量+封驳判定：
```bash
python {SKILL_DIR}/scripts/case_json_manager.py analyze --file {TC_PATH} --reqs F001-F054 --output {PROJECT_DIR}/.tmp/analysis.json
```

### 步骤 2：覆盖度分析

**需求覆盖率** = 已覆盖需求数 / 总需求数 × 100%
**场景覆盖率** = 已覆盖场景数 / (需求数 × 3) × 100%

每个需求检查：正常流程 ✅ / 异常流程 ✅ / 边界条件 ✅

### 步骤 3：质量检查

检查项及判定标准：

| 检查项 | 通过标准 | 不通过示例 |
|--------|---------|-----------|
| 标题清晰 | 明确一个测试点 | "验证登录功能"（太粗） |
| 前置条件 | 包含测试数据和环境 | "系统正常运行"（缺具体数据） |
| 步骤可执行 | 动词+对象+具体数据 | "输入正确数据"（无具体值） |
| 预期可验证 | 含具体元素/文字/数值 | "显示成功"（不可验证） |
| 优先级合理 | 核心流程=高 | 正常登录标为"低" |
| 关联正确 | req字段对应正确需求 | TC003测登录但req=F005 |

### 步骤 4：封驳判定

**封驳条件**（满足任一）：需求覆盖率<85% / P0/P1需求未覆盖 / 步骤不可执行>5个 / 预期不明确>5个 / 关键场景遗漏

结论：✅通过（≥95%无严重问题）/ ⚠️有条件通过（≥90%仅改进建议）/ 🚫封驳

### 步骤 6：输出结论

通过 → 生成 XLSX + 告知用户结论和下一步
有条件通过 → 展示改进点，用户可选择补充或直接执行
封驳 → 展示必须修复的问题（含 index），引导重新评审

## 评审标准

需求覆盖率：≥95%通过 / ≥90%有条件 / <85%封驳
场景完整性：每需求至少 正常+异常+边界
用例有效性：步骤清晰度100%、预期明确度100%
冗余率：≤10%
