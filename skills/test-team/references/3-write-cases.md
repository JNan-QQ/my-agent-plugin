
# 测试用例编写 Agent

## 测试设计方法

### 等价类划分

将输入域分为有效/无效组，每组取代表值。

**示例（登录密码字段）**：
| 等价类 | 代表值 | 用例 |
|--------|--------|------|
| 有效：6-20位字母数字 | `Abc123` | TC001 正常登录 |
| 无效：空值 | `""` | TC002 空密码提示 |
| 无效：<6位 | `Ab1` | TC003 密码过短提示 |
| 无效：>20位 | `A1*20+` | TC004 密码过长提示 |

### 边界值分析

取边界点及其两侧值。

**示例（密码长度6-20）**：5位(无效)、6位(有效)、7位(有效)、19位(有效)、20位(有效)、21位(无效)

### 场景法

基于用户操作路径设计端到端场景，每场景串联多个功能点。

**示例（电商下单）**：浏览商品 → 加入购物车 → 填写地址 → 选择支付方式 → 提交订单 → 支付成功

### 错误推测法

基于经验推测高频出错点：并发操作、网络中断、超长输入、特殊字符、权限边界。

## 工作模式

- **模式1 全新生成**（默认）：基于需求从零设计用例
- **模式2 增量优化**：读取评审报告，针对性修改

## 工作流程

### 步骤 0：检测模式

Glob `**/case-review-*.md`（path: 工作目录）。存在评审报告 → 询问用户选择模式。

### 步骤 1：读取需求

Glob `**/REQ-*.md`（path: 工作目录），优先选已通过评审的需求文档。

增量模式额外读取：case-review-*.md（提取遗漏/问题/冗余）+ TC-*.json

### 步骤 2：设计用例

#### 全新模式：每个需求设计用例

每个需求至少覆盖 3 类用例：

| 类型 | 覆盖内容 | 示例（登录功能） |
|------|---------|----------------|
| 正常流程 | 基本功能、有效输入 | 正确用户名+密码 → 登录成功 |
| 异常流程 | 错误输入、异常情况 | 错误密码 → 提示"用户名或密码错误" |
| 边界条件 | 最小/最大值、空值、特殊字符 | 密码含SQL注入字符 → 正常拒绝无报错 |

#### 增量模式：按评审报告修改

- A. 补充遗漏：按"遗漏场景"新增，编号继续递增
- B. 修改问题：按 index 定位，修改步骤/预期
- C. 删除冗余：按 index 标记删除
- D. 调整优先级：按建议修改

### 步骤 3：生成 JSON 用例

保存到 `test-output/test-cases/TC-YYYYMMDD-HHMM.json`

```bash
# 全新
python {SKILL_DIR}/scripts/case_json_manager.py create --output {OUTPUT_PATH} --data test-output/.tmp/cases.json
# 增量
python {SKILL_DIR}/scripts/case_json_manager.py update --file {TC_PATH} --index 0 --data test-output/.tmp/update.json
python {SKILL_DIR}/scripts/case_json_manager.py append --file {TC_PATH} --data test-output/.tmp/new.json
python {SKILL_DIR}/scripts/case_json_manager.py delete --file {TC_PATH} --indices 0,1,2
```

**JSON 字段**：id(TC001) / title / precondition / steps(\n分隔) / expected / priority(高/中/低) / type / req(F001) / note

**JSON 格式规则（必须遵守）**：
1. 字符串中的双引号必须转义为 `\"`（如：`输入 \"test\" 密码`）
2. 字符串中的换行用 `\n`，不要用真实换行符
3. 生成后必须用 Write 工具写入文件，不要只输出到聊天
4. 写入后验证：读取文件前几行确认 JSON 格式正确

### 步骤 4：输出统计

全新模式：用例总数 + 按优先级/类型分布 + 需求覆盖率
增量模式：新增/修改/删除数 + 优化详情

## 用例去重规则（必须遵守）

用例雷同的根源是测试点模糊。以下规则确保每条用例唯一：

1. **一条用例 = 一个测试点**：标题必须明确"验证什么"，如"验证空密码登录提示错误"
2. **去重检查**：设计完一批后，逐对比 title 字段，如果两条用例的测试点相同，合并或删除一条
3. **区分测试点粒度**：
   - ❌ "验证登录功能"（太粗，包含多个测试点）
   - ✅ "验证正确密码登录成功"、"验证错误密码提示错误"、"验证空用户名提示错误"
4. **正交覆盖**：正常/异常/边界三个维度的用例不应有交集——如果一条"异常用例"也能被"正常用例"覆盖，删掉它

### 去重示例

**❌ 雷同用例（3条测同一件事）**：
- TC001: 输入错误密码，验证登录失败
- TC002: 密码不正确时，验证无法登录
- TC003: 输入无效密码，验证提示错误

**✅ 去重后（3条各有不同测试点）**：
- TC001: 输入错误密码，验证提示"用户名或密码错误"
- TC002: 密码为空，验证提示"请输入密码"
- TC003: 密码连续错误5次，验证账号锁定30分钟

## 步骤与预期结果写法规范

### 步骤（steps）规范

每步一个操作，使用"动词+对象"格式：

| ❌ 模糊写法 | ✅ 清晰写法 |
|------------|-----------|
| 操作登录页面 | 1. 打开登录页面 http://example.com/login |
| 填写信息 | 2. 在用户名输入框输入 "admin" |
| 测试各种情况 | 3. 在密码输入框输入 "wrong123" |
| 检查结果 | 4. 点击"登录"按钮 |

**规则**：
- 每步只包含一个用户操作（点击/输入/选择/等待）
- 涉及页面元素时写出元素名称（按钮文字、输入框标签）
- 涉及数据时写出具体值（不要写"输入正确密码"，要写"输入 `Abc123`"）
- 步骤数 ≤ 10 步

### 预期结果（expected）规范

预期结果必须可验证——测试人员看完就知道是否通过：

| ❌ 模糊写法 | ✅ 可验证写法 |
|------------|-------------|
| 显示错误信息 | 页面顶部显示红色提示"用户名或密码错误" |
| 跳转成功 | 跳转到首页 http://example.com/dashboard，显示用户名"admin" |
| 数据保存成功 | 数据库 users 表新增一条记录，name="test" |
| 性能良好 | 接口响应时间 < 2s |

**规则**：
- 包含具体的页面元素（文字/位置/样式）
- 包含具体的数值/状态/URL
- 不使用"正常"、"成功"、"正确"等模糊词——必须描述具体表现

## 设计原则

1. 单一职责：每用例验证一个测试点
2. 步骤清晰：≤10步，任何人可执行
3. 结果明确：预期可验证
4. 优先级合理：核心功能=高
5. 可追溯：关联需求编号

## 质量标准

- 需求覆盖率 100%
- 每需求至少：正常+异常+边界
- 步骤≤10步，预期可验证
- 编号连续无重复
- 无雷同用例（去重检查通过）

## 用例编写反例

| ❌ 错误做法 | ✅ 正确做法 | 原因 |
|------------|-----------|------|
| 3条用例都测"密码错误" | 分别测：错误密码、空密码、锁定后密码 | 测试点重复 |
| 步骤写"输入正确数据" | 步骤写"输入用户名 `admin`，密码 `Abc123`" | 数据不具体无法执行 |
| 预期写"显示成功" | 预期写"跳转到首页，顶部显示'欢迎 admin'" | 不可验证 |
| 1条用例覆盖登录+注册+找回密码 | 拆成3条，每条覆盖一个功能 | 粒度过粗 |
| 预期写"系统不报错" | 预期写"页面无红色提示，控制台无Error日志" | 消极验证不可靠 |
